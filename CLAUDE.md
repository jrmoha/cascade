# CLAUDE.md — Cascade

Real-time event analytics platform. This file is the brief; the full reasoning lives in `docs/00-charter.md`, `docs/blueprint.md`, and `docs/adr/`. Read those before non-trivial work.

## What this is

A self-hostable engine that ingests high-volume events, stores them raw, computes real-time aggregations (counters, funnels, retention, leaderboards), and serves them. Demo domain: game telemetry. Backend + deployment + scaling only — no frontend.

## Architecture (see docs/adr/0001)

Event-driven. Collector → **Kafka** → two independent consumers: Ingestion-Processor (writes raw to **Cassandra**) and Aggregator (derives read models). Query API serves **only** from read models. **PostgreSQL** holds metadata; **Redis** holds hot counters/leaderboards/rate-limits. Services: Collector, Ingestion-Processor, Aggregator, Query API, Project/Schema.

## Non-negotiable rules

- **Write path and read path are separate; analytics never scans raw.** Counters, funnels, retention and leaderboards are served **only** from Aggregator read models — never by aggregating over raw Cassandra. The Aggregator (a second, independent `raw-events` consumer; group `cascade-aggregator`) is the **only** writer of read models and picks a store per access pattern (ADR-0015): **time-series counters → Cassandra** _aggregate_ tables, **ranked top-N → Redis** sorted sets, **funnel/retention summaries → Postgres** (its **own** tables via raw `pg` + a SQL migrator, kept separate from Project/Schema's Prisma schema). It windows on **event time (`occurredAt`)**, stays idempotent by deduping on `eventId` before additive ops, and every view is **rebuildable** by replaying `raw-events` from offset 0. The invariant: the Aggregator never touches the **raw write path** (`raw_events`) — that broadens ADR-0001's "must not touch Cassandra" to "must not touch the raw write path". (As of KAN-32 the Aggregator derives its first view: per-`(project, type, time-bucket)` **event counts** — minute + hour Cassandra `counter` tables, windowed on `occurredAt`, made replay-safe by a per-`eventId` Redis dedup gate (`AGGREGATOR_DEDUP_TTL_SECONDS`) before each `+1`. As of KAN-34 it also derives a per-project **live leaderboard** in **Redis sorted sets** — keyed `lb:{projectId}:{period}` (`period` = `alltime` or a UTC `YYYY-MM-DD`; the daily board is TTL'd by `AGGREGATOR_LEADERBOARD_DAILY_TTL_SECONDS`), member = `payload.playerId`, score = `payload.score`, applied with **best-score `ZADD GT`** so it is naturally idempotent/replay-safe with **no dedup needed** (ADR-0016 §1); the **Query API** serves top-N + a player's rank from it (`GET /leaderboard`, `GET /leaderboard/rank`) and now has Redis as a readiness dep. As of KAN-35 (ADR-0017) it also derives per-actor **funnel** and **retention** views in **Postgres** (its own tables via the `pg` SQL migrator): `funnel_actor_steps(project_id, actor_id, event_type, first_seen_at)` upserted with **`LEAST`** (earliest first-seen) and `retention_actor_activity(project_id, actor_id, active_period)` inserted with **`ON CONFLICT DO NOTHING`** (active-day set) — both **naturally idempotent/replay-safe with no dedup** (ADR-0016 §1), keyed on the actor identity `actorId ?? sessionId` (shared `actorKey()` helper; events with neither are skipped). The **Query API** computes them at read time from Postgres — `GET /funnel?projectId=&steps=&from=&to=` (ordered conversion over an ad-hoc 2–10 step list; furthest-step is derived, not stored) and `GET /retention?projectId=&from=&to=&maxOffset=` (cohort = `MIN(active_period)` per actor) — and now has **Postgres** as a readiness dep (read-only; the Aggregator owns the schema). As of KAN-36 (ADR-0018) the **counts** view finally gets its read endpoint — `GET /counts?projectId=&from=&to=&granularity=minute|hour&type=` serves a per-`(bucket, eventType)` time-series straight from the `event_counts_by_minute`/`_by_hour` counter tables (bucket-walk single-partition SELECTs, span-capped: `hour` ≤ `MAX_QUERY_BUCKETS`=168, `minute` ≤ `MAX_COUNTS_MINUTE_BUCKETS`=1440), so all four analytics surfaces (counts, leaderboard, funnel, retention) are now live and boundary-clean. **The CQRS read boundary is now enforced, not just prose:** `services/query-api/test/cqrs-boundary.spec.ts` asserts the analytics modules never reference `raw_events`/`RawEventReadRepository` and that `raw_events` appears **only** under `src/query/` (the sanctioned ADR-0008 retrieval path) — the check targets the raw **write path**, not Cassandra in general (counts legitimately reads Cassandra _aggregate_ tables). All analytics reads are **eventually consistent** (they reflect only what the Aggregator has processed — documented as expected behaviour, not a bug). See `docs/read-models/event-counts.md`, `leaderboard.md`, `funnel.md`, and `retention.md`.) **KAN-33 (the keystone, ADR-0016)** pins down and proves the idempotency/replay contract: offsets commit only **after** the durable write (`ServerKafka` `eachMessage` resolves the offset post-handler; `autoCommit` is explicit in `main.ts`), a replay from offset 0 into truncated tables **with the dedup keyspace flushed** reproduces aggregates byte-for-byte, and the accepted residual edge (crash between `SETNX` and the write → one lost increment) is healed by rebuild — "effectively-once" is the honest target. Proven in `services/aggregator/test/idempotency.e2e-spec.ts` (out-of-order + rebuild + restart). If a feature needs to _aggregate_ over raw rows live, that's a design smell; stop and flag it. **Bounded raw _retrieval_** is allowed and supported: the Query API's `GET /query?projectId=&from=&to=` reads raw events for replay/audit/debugging, but only ever **partition-key-bounded** (per-`(project_id, time_bucket)` single-partition slices, span-capped) — never a cross-partition scan, never `ALLOW FILTERING`. See ADR-0008 (supersedes the Phase-0 ADR-0003), ADR-0015, and `docs/contracts/events.md` → "Reading events back".
- **Cassandra is modeled query-first.** Never add a table without stating the exact query it serves and its partition key. Watch partition size; bucket by `project_id + time_bucket` (hourly). Schema changes go through the versioned migrations in `services/ingestion-processor/migrations/` (applied by `Migrator` on startup and `npm run migrate`) — never ad-hoc `cqlsh` or inline `CREATE TABLE`. See ADR-0007 for the `raw_events` model. **Replication & consistency are fixed by ADR-0019 (KAN-37):** Cassandra is `NetworkTopologyStrategy` with read/write `LOCAL_QUORUM` (`R+W>RF` ⇒ strong per-DC), and Postgres is a primary + async read-replica with read-after-write/config reads on the primary and eventually-consistent analytics reads on the replica. **KAN-38 landed the Cassandra piece:** the keyspace is created NTS with RF from `CASSANDRA_REPLICATION_FACTOR` (3 in the compose cluster, 1 single-node dev/test) and all three Cassandra clients set `queryOptions.consistency` from `CASSANDRA_CONSISTENCY` (never the driver default `LOCAL_ONE`) — both validated via the shared `cassandraConsistencySchema` in `@cascade/contracts`; `infra/docker-compose.yml` runs a 3-node cluster (see `docs/runbooks/cassandra-cluster.md` + `infra/scripts/consistency-demo.sh`). Integration/smoke tests stay single-node (RF=1). **KAN-39 proved node-loss survival** (the chaos gate): under continuous `cassandra-stress` load at `LOCAL_QUORUM`, killing one of the three nodes yields **0 errors** (quorum = 2 of 3), the node rejoins and catches up via hinted handoff / repair, and a **second** node down deterministically returns `Unavailable` — the chosen CAP limit, not a bug. Repeatable via `infra/scripts/node-down-chaos.sh` (shares `infra/scripts/cassandra-lib.sh` with the consistency demo); see `docs/runbooks/cassandra-node-down.md`. **KAN-41 landed the Postgres piece:** `infra/docker-compose.yml` runs a **primary** (`cascade-postgres-primary`, `:5432`) + one **streaming async read replica** (`cascade-postgres-replica`, `:5433`), hand-rolled on the official image (primary initdb hook creates the `replicator` role + pg_hba rule; replica bootstraps via `pg_basebackup -R` and runs as a hot standby — `infra/postgres/{primary,replica}/`). Read routing is **by freshness (ADR-0019 §2), not blindly**: **Project/Schema (Prisma) and the Aggregator stay 100% on the primary** (config-critical / read-your-writes and write-only), while the **Query API funnel/retention** analytics reads route to the replica via a second `pg` pool (`DATABASE_REPLICA_URL` → `PostgresService.replicaQuery`) — **unset ⇒ falls back to the primary**, so single-node dev/test and the smoke test are unchanged. No Prisma read-replica extension. Replication + the read-only-standby failure boundary are proven by `infra/scripts/postgres-replication-demo.sh` (`make pg-replication-demo`); manual failover is documented, not automated. See `docs/runbooks/postgres-replication.md`.
- **Postgres (Project/Schema service) is the one ORM exception — Prisma, fenced to persistence.** The relational config store (projects, API keys, event schemas) uses **Prisma** over Postgres; everywhere else drivers are used raw. Prisma owns only the **DB schema** (`services/project-schema/prisma/schema.prisma`) via **versioned migrations** under `prisma/migrations/` — applied with `prisma migrate deploy` on service boot and `npm run migrate`, never `db push`/`synchronize`. The **wire/API contracts stay Zod** in `@cascade/contracts` (Prisma must not leak into the cross-service surface). API keys are stored **argon2-hashed** with a unique-indexed non-secret `prefix` for one-lookup verification, and are revocable; event schemas are JSON Schema in a `jsonb` column. See ADR-0011 and `docs/contracts/project-schema.md`.
- **All Kafka consumers must be idempotent.** Delivery is at-least-once. Re-processing the same message must not double-count. Consumers must not throw out of the handler (a poison message would loop forever and block its partition): route failures to the dead-letter topic instead — validation failures immediately, transient (persistence) failures after a bounded retry. See ADR-0006 / `docs/runbooks/dlq.md`.
- **No new cross-service synchronous call without justification.** Prefer events. Sync calls (gRPC/REST) only for genuine request/response needs. Service boundaries (one responsibility each), the topic inventory, and the sync-call inventory are recorded in ADR-0009 — a new cross-service interaction is "real" only once it's listed there. **Kafka topics are lowercase-hyphen with a `.dlq` suffix for dead letters** (`raw-events`, `raw-events.dlq`), defined as constants in `@cascade/contracts` and imported — never re-typed as string literals. **Topics are provisioned explicitly, never auto-created (KAN-40, ADR-0020):** `raw-events` is **6 partitions / RF=3 / min.insync=2** on a **3-broker** cluster (`kafka-1/2/3`), created by the one-shot `kafka-init` service (auto-create is OFF); the partition key is **`sessionId ?? actorId ?? eventId`** (per-session ordering, spreads a hot project so consumers scale — the DLQ stays `projectId`-keyed). The Ingestion-Processor and Aggregator run as **multiple instances per consumer group** (`make stack-scale`; partitions distribute across them, rebalance-safe because writes are idempotent per ADR-0016). Producers use `acks=all` so RF=3 + min.insync=2 survives one broker down. **Gotcha:** NestJS `ServerKafka` postfixes broker-side group ids with `-server` (`cascade-ingestion-processor-server`). See `docs/runbooks/kafka-scaling.md` + `infra/scripts/kafka-scaling-demo.sh`. The **one** justified sync call (Collector → Project/Schema, `VerifyKey`/`GetEventSchema`) is **gRPC**: the contract is `libs/contracts/proto/project_schema.proto`, ts-proto-generated into `libs/contracts/src/generated/` (committed; re-exported as the `projectSchemaProto` namespace) by `npm run proto:gen` — the generated type **is** the contract, imported by both sides. Project/Schema is a hybrid HTTP + gRPC app; REST stays its admin surface. As of KAN-30 this call is **live**: the Collector's ingest path authenticates an **`x-api-key`** header (the `projectId` is **derived from the key**, never the request body — `collectEventSchema` omits it) and validates each `payload` against the project's registered JSON Schema (Ajv, compiled once). Check order is **key (`401`) → envelope (`400`) → per-project schema (unregistered type `422`, bad payload `400`)**. Key→projectId and schemas are **Redis-cached** (short TTL, `PROJECT_SCHEMA_CACHE_TTL_SECONDS`; key hashed before use as a cache key); a cache hit makes no gRPC call. The policy when Project/Schema is unreachable on a cold cache is **fail-closed (`503`)** — never admit an unauthenticated/unvalidated event. Redis is a Collector readiness dep; Project/Schema is not. See ADR-0012 and ADR-0013.
- **Cross-service contracts are versioned and compatibility is CI-enforced.** Rule everywhere: **additive is safe; rename/remove/retype is breaking** (consumers tolerate unknown fields, never depend on field order). The `raw-events` envelope carries a server-stamped integer `schemaVersion` (`RAW_EVENT_SCHEMA_VERSION`); bump it only for a breaking change, never for additive. proto field numbers are part of the contract — never reuse or renumber. A breaking change **fails CI** (`.github/workflows/ci.yml`): the JSON-Schema contract snapshot test (`libs/contracts/test/contract-snapshot.spec.ts`, regenerate with `vitest -u`), `npm run proto:check` (regenerate + git-diff `src/generated`), and `tsc --build`. The committed `src/generated/` is lint/format-ignored to keep `proto:check`'s diff stable. See ADR-0012.

## Stack & conventions

- NestJS 11 + TypeScript. Node 22+ (LTS). (Pin versions in package.json.) Postgres access in the Project/Schema service is **Prisma 7**, which has no embedded query engine — it runs through the `@prisma/adapter-pg` driver adapter, and the Migrate datasource URL lives in `services/project-schema/prisma.config.ts` (not `schema.prisma`). See ADR-0014.
- Lint/format: ESLint + Prettier. Commits: Conventional Commits (Commitlint + Husky).
- Tests: Vitest (unit), Testcontainers (integration against real Cassandra/Kafka/Postgres). Don't mock the database in integration tests. NestJS services need `unplugin-swc` in their `vitest.config.ts` so decorator metadata is emitted for DI (see `services/collector`). Gate Docker-dependent integration tests behind `SKIP_INTEGRATION=1`.
- Monorepo: `services/`, `libs/` (shared contracts), `infra/` (Terraform + docker-compose), `docs/`. Each service is **independently deployable**: its own multi-stage `Dockerfile` (build context = repo root so `@cascade/contracts` resolves), added to `infra/docker-compose.yml` under the `apps` profile (`make up` = infra only; `make stack-up` = full stack). See ADR-0010.
- **Config is env-only, validated with Zod at boot — no inline defaults.** Each service has a per-service Zod env schema (`src/config/env.schema.ts`) parsed once at startup into a frozen, typed `APP_CONFIG`; a missing/invalid var fails fast (12-factor). Infra/peer addresses (`KAFKA_BOOTSTRAP_SERVERS`, `CASSANDRA_*`, the Project/Schema service's `DATABASE_URL`) are **required** and come from config via container service names — never hardcoded, never `?? 'localhost'`. Only a service's own HTTP `PORT` keeps a default. Document every var in `services/<svc>/.env.example`.
- **Every service exposes `GET /health` (liveness) and `GET /ready` (readiness)** via `@nestjs/terminus` — readiness pings its deps (Kafka and/or Cassandra) and returns `503` when one is down. The Ingestion-Processor is a **hybrid app** (`NestFactory.create` + `connectMicroservice` + `startAllMicroservices`): a Kafka consumer plus a small HTTP server for the probes. See ADR-0010.
- **Shared contracts use Zod as the single source of truth.** Define the schema in `@cascade/contracts` and derive the TS type via `z.infer` (never hand-write a parallel `interface` + validator — they drift). The canonical event envelope is `rawEventSchema`/`RawEvent` (see ADR-0004, `docs/contracts/events.md`). It separates `occurredAt` (event time, from the client) from `receivedAt` (ingest time, stamped by the Collector); Cassandra `time_window` buckets by `occurredAt`. It also carries a server-stamped `schemaVersion` (wire-only — not persisted to Cassandra; see ADR-0012). The Collector validates before producing and the Ingestion-Processor validates on consume. HTTP-edge validation derives its schema from the contract (`collectEventSchema` = `rawEventSchema.omit(...).strip()`) rather than re-implementing it; invalid events get a structured `400` and never reach Kafka (ADR-0005).

## Local env gotchas

- Cassandra needs a healthcheck + startup wait; don't connect before it's ready.
- Cassandra datacenter is **`datacenter1`** (`GossipingPropertyFileSnitch`, honouring `CASSANDRA_DC`). Clients must set `localDataCenter`/`CASSANDRA_LOCAL_DC=datacenter1`, or the driver finds no hosts.
- **`make up` starts a 3-node Cassandra cluster** (`cascade-cassandra-1/2/3`, KAN-38): slower first boot (nodes join sequentially), heavier RAM (trim Docker if needed), and every Cassandra service needs `CASSANDRA_REPLICATION_FACTOR` + `CASSANDRA_CONSISTENCY` set. Verify with `docker exec cascade-cassandra-1 nodetool status` (3× UN). See `docs/runbooks/cassandra-cluster.md`.
- **`make up` also starts a 3-broker Kafka cluster** (`cascade-kafka-1/2/3`, KAN-40) + a one-shot `cascade-kafka-init` that provisions topics and exits 0; app services gate on it (`service_completed_successfully`). Host ports differ per broker (`localhost:9092/9094/9095`); internal is `kafka-1:29092,kafka-2:29092,kafka-3:29092`. Verify topics with `docker exec cascade-kafka-1 /opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:29092 --describe --topic raw-events` (6 partitions, RF=3). Scale consumers with `make stack-scale`. See `docs/runbooks/kafka-scaling.md`.
- **`make up` also starts a Postgres primary + read replica** (`cascade-postgres-primary` on `:5432`, `cascade-postgres-replica` on `:5433`, KAN-41): the replica `pg_basebackup`s the primary before it reports healthy (slower first boot, two PG instances — trim Docker RAM if needed). Verify streaming with `docker exec cascade-postgres-primary psql -U cascade -d cascade -c "SELECT application_name,state FROM pg_stat_replication;"` (one `streaming` row) and `docker exec cascade-postgres-replica psql -U cascade -d cascade -tAc "SELECT pg_is_in_recovery();"` (`t`). The replica is **read-only** — a write against `:5433` errors "cannot execute … in a read-only transaction" (working as designed). `make down-v` wipes both volumes; the replica re-bootstraps on next `make up`. Demo: `make pg-replication-demo`. See `docs/runbooks/postgres-replication.md`.
- Kafka listener config matters: advertise a host-reachable listener for tooling AND an internal listener for containers. Use the container service name (not `localhost`) for service-to-service; use `127.0.0.1:<mapped-port>` from the host.

## Working agreement (IMPORTANT)

- **Plan first.** For any ticket, produce a short plan (files to touch, approach, tradeoffs) and wait for approval before writing code. No code in the planning step.
- **One Jira ticket at a time.** Reference the ticket key (e.g. KAN-16) in the branch and commit.
- **Document after implementing.** When a ticket is done: update the relevant `docs/` page, add/append an ADR if a real decision was made, and update this CLAUDE.md if a convention changed.
- **Definition of done:** code + passing tests + docs updated + Jira ticket moved to Done.

## Phase gate

Currently in **Phase 0 — walking skeleton** (epic KAN-5). Do not start Phase 1 work until the end-to-end smoke test (KAN-20) passes.

The gate is `npm run smoke` (the `@cascade/e2e` workspace; see README → "End-to-end smoke test"). It stands up real Kafka + Cassandra (and, since KAN-30, Postgres + Redis) via Testcontainers, boots the services in-process, and asserts an authenticated event round-trips Collector → (Project/Schema, cached) → Kafka → Cassandra → Query API. **As of KAN-20 this passes** — Phase 1 work may begin. Keep it green; treat a red smoke test as a release blocker.
